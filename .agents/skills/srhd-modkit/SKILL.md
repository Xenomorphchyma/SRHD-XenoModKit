# SRHD ModKit

Использовать SRHD ModKit как основной программный интерфейс к модам Space Rangers HD. Работать через CLI или Python API без GUI.

## Найти корень ModKit

1. Найти корень репозитория, содержащий `srhd.py`, `pyproject.toml` и каталог `srhd_modkit`. Обозначать его как `<MODKIT_ROOT>`.
2. Если этот скилл загружен прямо из репозитория, `<MODKIT_ROOT>` находится на три уровня выше `SKILL.md`: `.agents/skills/srhd-modkit/../../..`.
3. Выполнить `python -B srhd.py --version` из `<MODKIT_ROOT>`. Ожидать `SRHD ModKit 0.9.5` или новее.
4. При другой версии сначала прочитать локальные `README_RU.md`, `AUDIT_RU.md` и вывод `python -B srhd.py --help`; не предполагать наличие `script decompile`.
5. Если корень найти нельзя, запросить путь к клону ModKit. Не использовать каталог установленной игры вместо инструментария.
6. Для точного синтаксиса читать [references/commands.md](references/commands.md).

## Выбирать workflow по задаче

- Для первичного анализа запускать `audit --profile dev --json`.
- Для проверки коллекции передавать корень коллекции той же команде; ModKit сам обнаружит моды.
- Для DAT использовать только `dat`-команды. Сначала читать дерево или значение, затем писать в новый DAT и повторно запускать `dat validate` или релизный аудит.
- Для скриптов сначала выполнять `script audit-mod` и `script lint-runtime`; если исходника нет, восстанавливать его через `script decompile`, затем изменять RSON headless-командами и собирать через `script build`. Для доказательства различий двух бинарников использовать `script compare-scr`, а не ручный поиск байтов. Если импорт непустого Lang.dat падает, изучить `lang_import.diagnostic`; продолжать с `--fallback-without-lang` только по явному решению, потому что диалоги не будут восстановлены.
- В скриптах не сохранять сырые `Item`, `Planet`, `Star` или `Ship` в долгоживущем `TVar`/массиве: сохранять `Id(object)` и при миграции сначала обнулять старую ссылку. Для `Planet`/`Ship` использовать подтверждённые `IdToPlanet`/`IdToShip`. Не считать `IdToStar` встроенной функцией: звезду восстанавливать локальным ограниченным обходом `GalaxyStars()`/`GalaxyStar(i)` либо через сохранённую планету. Результат `GalaxyStar` отдельно проверять до `Star*`, `Id`, `RelationToRanger` и `Dist`; не передавать прямой `GalaxyStar(...)` в `Dist`. До `ShipStar(ship)` отдельно обработать посадку и доказать `ShipInNormalSpace(ship)` вместе с `!ShipIsTakeoff(ship)`; guard и объектный вызов разносить по операторам, не рассчитывая на короткое замыкание `&&`/`||`. `OrderTakeOff` и дальнейшие пространственные запросы разделять границей хода. Raw handle из `ShipGetBad` не разыменовывать: сначала сопоставить его с живым кораблём из текущей системы, а элемент `StarShips` перед `ShipIsTakeoff` дополнительно ограничить `ShipTypeN < t_RC`, исключая станции. Разгрузку и `ShipOut` также разделять границей хода. Никогда не выполнять `ShipOut` текущего `GroupShip` в прямом цикле с повторным `GroupCount`, даже с уменьшением индекса; использовать отдельный обратный проход и завершать обработчик после изменения группы. Persistent-массив инициализировать через `newarray(...)`; новый массив мигрировать под отдельной версией хранилища, а не под уже пройденным first-run gate. После `newarray(1)`/`ArrayClear` начинать записи с `1`, пустоту проверять через `ArrayDim <= 1`, а парные persistent-массивы — на равенство размеров. В RSON держать не более четырёх `TGroup`, не обращаться из диалога к persistent-массиву, расположенному позднее handler, и не повторять локальные имена в разных ветвях одного RScript handler/function. Не передавать Item третьим аргументом `RndObject` и не менять `Order*`/`ShipSetBad` до проверки `ShipInHyperSpace`.
- Для GI/PNG использовать нативные `convert gi-png` и `convert png-gi`; RangerTools и GUI не нужны. Для остальных ресурсов использовать `resource info/list/verify/extract/build-gai/build-pkg`. Не пытаться записывать HAI.
- Для текстового квеста сначала выполнять `quest info` и `quest validate`: изучать `location_images`, локации без кадров и `image_usage`. Для всего мода запускать release-аудит и проверять отдельные `quest-cards`, `quest-media`, регистрацию `Bm.PQI.*`, неиспользуемые ассеты и параметры `343x394 RGB`. Редактировать через `quest export-json`, собирать новым путём через `quest build`, затем выполнять `quest roundtrip`. TGE и GUI не нужны. Не объявлять визуальное постоянство персонажа или предмета доказанным статическим анализом.
- Для активного набора модов использовать `compat` с существующим ModCFG только для чтения.
- Для публикации использовать `release check`, затем `release build`. Не считать низкоуровневый `pack` полноценной релизной проверкой.
- При подозрении на оставшиеся RScript/BlockParEditor сначала выполнять `doctor processes`; `--terminate` использовать только после просмотра отчёта.
- Если CLI недоступен, пользователь может завершить редактор по исходному имени EXE во вкладке «Подробности» Диспетчера задач; процесс не является защищённым или скрытым от системного списка.

## Интерпретировать отчёт честно

- Считать `failed` сбоем самой проверки, а не ошибкой мода.
- Считать `unsupported` неполным покрытием: сохранять файл побайтно, не объявлять его корректным или редактируемым.
- Считать `issues` контейнером ошибок, предупреждений и сведений; проверять `severity` каждого элемента.
- Учитывать, что профиль `dev` быстрый и неполный, а `release` декодирует все DAT и глубоко проверяет поддерживаемые ресурсы.
- Не называть победителя коллизии при `resolution: unknown`.
- Использовать `--allow CODE[:GLOB]` только для осознанного исключения с объяснением пользователю. Не скрывать поле `suppressed` из результата.

## Правила изменений

1. Сначала проверить состояние Git. Не перезаписывать и не откатывать посторонние пользовательские изменения.
2. Не редактировать установленные игровые файлы для диагностики. Создавать staging или рабочую копию вне каталога игры.
3. Не менять `Mods/ModCFG.txt`; `compat` и `modcfg` использовать только для чтения.
4. Не запускать `open`, не выставлять `SRHD_MODKIT_ALLOW_GUI=1`, не управлять ResEditor, RScript или BlockPar через видимый интерфейс.
5. Перед записью предпочитать новый output. Использовать `--overwrite` только когда пользователь явно разрешил замену или операция идёт во временной рабочей копии.
6. После изменения запускать ближайшую структурную проверку и затем `audit --profile release`.
7. Для релиза располагать ZIP вне дерева мода. Передавать пользователю ZIP вместе с внешними `*.manifest.json` и `*.audit.json`.
8. Использовать штатную адаптивную политику RScript: малый проект получает 60 секунд без подтверждённого прогресса и общий предел от 600 секунд, крупный — больше по размеру, объектам и строкам кода. Явный предел задавать только при доказанной необходимости; значение `0` отключает оба ограничения, но синтаксический preflight остаётся обязательным.
9. Непроверенный результат декомпиляции сохранять только по отдельному явно названному `--keep-unverified`; не подменять им штатный output и игровой SCR.

## Границы гарантии

- Не обещать реальное runtime-поведение без запуска игры.
- Не обещать совместимость с сохранениями и любой комбинацией сторонних модов.
- Не считать SCR без RSON полностью семантически проверенным.
- Не считать структурный round-trip QM/QMM доказательством фактического прохождения квеста в игре.
- Не создавать writer HAI и не декодировать неизвестные VDO, CMAP, VO, MAP, OPT или RAW.
- Считать круговой проход GAI/PKG доказательством структуры, но не визуальной корректности в игре.

## Завершать задачу
